Introduction

Buffer overflow vulnerabilities have been a critical security issue since the early days of computing. Despite being well-understood for decades, they continue to be discovered in modern software. Understanding how buffer overflows work is essential for both offensive security professionals and developers who want to write secure code.

Understanding Memory Layout

Before diving into buffer overflows, it's crucial to understand how programs use memory. When a program runs, the operating system allocates memory segments for different purposes:

  • Text Segment: Contains the executable code (read-only)
  • Data Segment: Stores initialized global and static variables
  • BSS Segment: Stores uninitialized global and static variables
  • Heap: Dynamic memory allocation (grows upward)
  • Stack: Local variables and function call information (grows downward)

How Buffer Overflows Work

A buffer overflow occurs when a program writes more data to a buffer than it was allocated to hold. This extra data can overwrite adjacent memory, potentially corrupting data or altering the program's execution flow.

Vulnerable Code Example

#include <stdio.h>
#include <string.h>

void vulnerable_function(char *input) {
    char buffer[64];
    strcpy(buffer, input);  // No bounds checking!
    printf("You entered: %s\n", buffer);
}

int main(int argc, char *argv[]) {
    if (argc > 1) {
        vulnerable_function(argv[1]);
    }
    return 0;
}

In this example, strcpy() copies the input string into a 64-byte buffer without checking the length. If the input exceeds 64 bytes, it will overflow the buffer and overwrite adjacent memory on the stack.

The Stack Frame

When a function is called, a stack frame is created containing:

  1. Function parameters
  2. Return address (where to continue after function ends)
  3. Saved base pointer (EBP/RBP)
  4. Local variables

Exploitation Techniques

The classic buffer overflow exploitation involves overwriting the return address to redirect program execution to attacker-controlled code (shellcode).

Basic Exploitation Steps

  1. Fuzzing: Find the crash point by sending increasing amounts of data
  2. Control EIP/RIP: Determine the exact offset to overwrite the return address
  3. Find Bad Characters: Identify characters that break the exploit
  4. Find JMP ESP: Locate a reliable way to redirect execution
  5. Generate Shellcode: Create payload that avoids bad characters
  6. Exploit: Combine everything into a working exploit

Modern Protections

Modern systems implement several protections against buffer overflows:

  • ASLR (Address Space Layout Randomization): Randomizes memory addresses
  • DEP/NX (Data Execution Prevention): Prevents execution of code in data segments
  • Stack Canaries: Detect stack corruption before return
  • PIE (Position Independent Executables): Randomizes code segment location
  • RELRO (Relocation Read-Only): Protects the GOT from overwrites

Prevention and Mitigation

As a developer, you can prevent buffer overflows by:

  • Using safe functions like strncpy(), snprintf()
  • Implementing proper input validation and bounds checking
  • Using modern languages with memory safety (Rust, Go)
  • Enabling compiler protections (-fstack-protector, -D_FORTIFY_SOURCE=2)
  • Regular code audits and static analysis

The best defense is writing secure code from the start. Understanding how attacks work helps you build better defenses.
